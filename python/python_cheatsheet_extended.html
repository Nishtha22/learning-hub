Syntax & Semantics

Python syntax fundamentals
Python uses indentation to define code blocks — no curly braces. Every block starts with a colon (:) and must be consistently indented (4 spaces is standard). This makes Python readable but sensitive to whitespace errors.
Key ideas

Indentation defines blocks — mixing tabs and spaces causes IndentationError.

Statements end at the newline — no semicolons needed (though allowed).

Use \ to continue a long line, or wrap in parentheses (preferred).

Comments start with #. Docstrings use triple quotes.

Python style guide PEP 8 says 4 spaces per indent. Use snake_case for variables/functions, PascalCase for classes, UPPER_CASE for constants.
Code example
# this is a comment

# multi-line expression (use parens)
result = (1 + 2
         + 3 + 4)

# docstring
def add(a, b):
    """Add two numbers and return result."""
    return a + b

# multiple statements on one line (avoid)
x = 1; y = 2

# Python is case-sensitive
name = "alice"
Name = "bob"   # different variable!
Basic data types
Python has several built-in primitive types. Understanding each type's behaviour — especially mutability — is essential. Mutable objects can be changed in place. Immutable objects create new objects on every change.
Key ideas

int — unlimited precision integers. float — 64-bit decimal numbers.

str — immutable sequence of characters. bool — True or False (subclass of int).

bytes — immutable byte sequence. bytearray — mutable bytes.

complex — complex numbers: 3+4j. None — the null singleton.

Floating point arithmetic is not exact — 0.1 + 0.2 != 0.3 in Python. For financial or exact decimal math always use the decimal module or integer arithmetic.
Code example
# int — unlimited size
x = 10 ** 100   # googol — no overflow!

# float — IEEE 754 double
0.1 + 0.2   # 0.30000000000000004 (floating point!)
from decimal import Decimal
Decimal("0.1") + Decimal("0.2")  # exact: 0.3

# bool is a subclass of int
int(True)   # 1
int(False)  # 0
True + True  # 2

# type conversion
int("42")     # 42
float("3.14")  # 3.14
str(100)      # "100"
bool(0)       # False
bool("")      # False

Variables & Operators

Variables & assignment
Variables are names that point to objects in memory. Python uses reference semantics — variables don't hold values directly, they hold references to objects. This matters especially with mutable objects.
Key ideas

Variables are created on first assignment — no declaration needed.

id(x) gives the memory address of the object x points to.

Augmented assignment: +=, -=, *=, /=, //=, %=, **=.

del x removes the variable name (the object is garbage collected if no other references exist).

b = a for a mutable (list, dict) makes both names point to the SAME object. Use b = a.copy() or b = a[:] for a shallow copy, or copy.deepcopy(a) for a full deep copy.
Code example
# multiple assignment
x = y = z = 0
a, b = 1, 2
a, b = b, a     # swap
first, *rest, last = [1,2,3,4,5]
# first=1, rest=[2,3,4], last=5

# reference semantics with mutables
a = [1, 2, 3]
b = a           # same object!
b.append(4)
print(a)        # [1, 2, 3, 4]
b = a.copy()    # now independent

# augmented assignment
x = 10
x += 5    # x = 15
x **= 2   # x = 225
Operators
Python has arithmetic, comparison, logical, bitwise, membership, and identity operators. Understanding is vs == and short-circuit evaluation is critical for interview questions.
Key ideas

// floor division, % modulo, ** power — all higher priority than you'd expect.

is / is not — identity (same object). == / != — equality (same value).

in / not in — membership test. O(1) for sets/dicts, O(n) for lists.

and / or short-circuit: return the actual operand, not True/False.

x or default is a common Python pattern for fallback values. But watch out: if x is 0, '', or False, the default triggers even when x has a value! Use 'x if x is not None else default' for safer code.
Code example
# arithmetic
17 // 5   # 3   (floor)
17 % 5    # 2   (remainder)
2 ** 10   # 1024

# identity vs equality
a = [1, 2, 3]
b = [1, 2, 3]
a == b    # True  (same value)
a is b    # False (different objects)

# short-circuit evaluation
0 or "default"   # "default"
5 or "default"   # 5
None and boom()  # None (boom never called)

# walrus := (Python 3.8+)
if (n := len(data)) > 10:
    print(f"Large: {n}")

Conditions & Loops

Conditional statements
Python uses if/elif/else for branching. The match-case statement (Python 3.10+) is the modern pattern-matching alternative to long elif chains. Conditions can use any truthy/falsy value.
Key ideas

Falsy: 0, 0.0, '', [], {}, set(), None, False. Everything else is truthy.

elif chain is evaluated top to bottom — first match wins, rest are skipped.

Ternary: x if condition else y — use only for simple cases.

match-case supports patterns: literals, types, tuples, guards (if clause).

match-case is structural pattern matching — not just a switch statement. It can destructure lists, dicts, and objects. case [x, y] matches any 2-element list. Very powerful for parsing.
Code example
# standard if/elif/else
score = 85
if score >= 90:   grade = "A"
elif score >= 80: grade = "B"
else:             grade = "C"

# ternary
status = "pass" if score >= 60 else "fail"

# match-case (Python 3.10+)
match command:
    case "quit": quit()
    case "help": show_help()
    case ["go", direction]: go(direction)
    case {"action": act}: do(act)
    case _ if debug: log(command)
    case _: print("unknown")
Loops — for, while, comprehensions
Python's for loop iterates over any iterable. while loops for condition-based repetition. List/dict/set comprehensions replace most map/filter usage and are idiomatic Python.
Key ideas

enumerate(iterable, start=0) — index + value pairs.

zip(*iterables, strict=False) — stops at shortest. strict=True raises on mismatch.

loop else — runs if loop finishes without break. Great for search patterns.

Comprehension: [expr for x in it if cond] — list, {k:v}, {x}, (x)

Generator expression (x**2 for x in range(n)) is lazy — values computed on demand. Use inside sum(), any(), all(), max() etc. for memory efficiency: sum(x**2 for x in range(10**6)).
Code example
# enumerate with custom start
for i, v in enumerate(items, start=1):
    print(f"#{i}: {v}")

# zip — stops at shortest
for name, score in zip(names, scores):
    print(name, score)

# dict comprehension
sq = {x: x**2 for x in range(5)}
# {0:0, 1:1, 2:4, 3:9, 4:16}

# set comprehension
evens = {x for x in range(20) if x%2==0}

# loop else
for item in collection:
    if item == target: break
else:
    print("Not found")  # if no break

Collections

Lists & practical examples
Lists are ordered, mutable sequences. They are the most versatile data structure in Python. Key interview patterns: two-pointer, sliding window, stack (append/pop), queue (deque).
Key ideas

list.sort() sorts in-place. sorted(list) returns new sorted list.

list[-1] last item. list[1:4] slice. list[::-1] reversed copy.

Use deque from collections for O(1) append/pop from both ends.

bisect module for binary search on sorted lists — O(log n).

list.pop(0) is O(n) — it shifts all elements. Use collections.deque for O(1) pops from the left. For interview problems needing a queue, always use deque.
Code example
# common list patterns
stack = []
stack.append(1)    # push
stack.pop()        # pop from end O(1)

from collections import deque
q = deque()
q.append(1)        # enqueue
q.popleft()        # dequeue O(1)

# flatten nested list
nested = [[1,2],[3,4],[5]]
flat = [x for sub in nested for x in sub]

# sort by multiple keys
students = [("Alice",90),("Bob",85),("Eve",90)]
sorted(students, key=lambda s: (-s[1], s[0]))
# by score desc, then name asc

# remove duplicates preserving order
seen = set()
unique = [x for x in lst if not (x in seen or seen.add(x))]
Dicts, Sets & Tuples
Dicts are the most used data structure after lists. Sets provide O(1) lookup and powerful set operations. Tuples are immutable sequences often used as records or named tuples.
Key ideas

Dict insertion, lookup, deletion: all O(1) average case.

dict.setdefault(k, []) — set default if key absent, return value.

Counter(iterable) — count occurrences. most_common(n) top n.

OrderedDict — preserves insertion order (dict does too in 3.7+, but OrderedDict has extra methods like move_to_end).

Counter subtraction: c1 - c2 removes elements. Counter addition: c1 + c2 adds counts. Counter intersection: c1 & c2 keeps minimums. Very useful for sliding window problems.
Code example
from collections import Counter, OrderedDict

# Counter — count frequencies
words = ["a","b","a","c","a","b"]
c = Counter(words)
c.most_common(2)  # [('a',3),('b',2)]
c["a"]           # 3
c["z"]           # 0 (no KeyError!)

# dict merge (3.9+)
defaults = {"color": "red", "size": 10}
overrides = {"size": 20}
config = defaults | overrides
# {"color": "red", "size": 20}

# set operations
a = {1,2,3,4}
b = {3,4,5,6}
a & b   # {3,4}  common
a | b   # {1,2,3,4,5,6} all
a - b   # {1,2} only in a

Functions

Functions — basics & signatures
Functions are first-class objects in Python. They can be passed as arguments, returned from other functions, and stored in data structures. Understanding argument types is critical for interview questions.
Key ideas

Positional-only args (before /): def f(a, b, /, c) — a, b must be positional.

Keyword-only args (after *): def f(a, *, b) — b must be keyword.

*args = extra positional args as tuple. **kwargs = extra keyword args as dict.

Never use mutable objects as default values — use None and create inside.

Functions remember their default argument values at definition time, not call time. def f(x, lst=[]) — the SAME list object is reused across every call. This is one of Python's most infamous gotchas.
Code example
def func(pos_only, /, normal, *, kw_only):
    print(pos_only, normal, kw_only)

# *args and **kwargs
def flexible(*args, **kwargs):
    print(args)    # tuple
    print(kwargs)  # dict

# correct mutable default
def add_item(item, lst=None):
    if lst is None: lst = []
    lst.append(item)
    return lst

# type hints
def divide(a: float, b: float) -> float | None:
    return a/b if b != 0 else None

# unpack into call
args = (10, 2)
divide(*args)   # 5.0
Lambda, map, filter & functools
Lambda creates anonymous inline functions. map and filter are lazy — they return iterators. functools provides higher-order function tools: partial, reduce, wraps, lru_cache, cache.
Key ideas

lambda args: expr — single expression, implicit return, no statements.

map(fn, it) lazy iterator. filter(fn, it) lazy iterator — wrap in list() to materialise.

functools.partial(fn, *args) — fix some arguments, return new function.

functools.reduce(fn, it, init) — fold/accumulate left to right.

functools.cache (Python 3.9+) is lru_cache with maxsize=None — simpler to write. Use it for recursive memoization. The cache stores ALL results so memory can grow unbounded — be careful with large inputs.
Code example
from functools import partial, reduce, lru_cache

# partial — fix arguments
import operator
double = partial(operator.mul, 2)
double(5)   # 10

# reduce — running total
reduce(lambda acc, x: acc+x, [1,2,3,4], 0)
# 10

# lru_cache — memoisation
@lru_cache(maxsize=None)
def fib(n):
    if n < 2: return n
    return fib(n-1) + fib(n-2)

# sorted with operator
import operator
sorted(people, key=operator.itemgetter(1))
Closures & decorators
A closure is a function that captures its enclosing scope's variables. Decorators use closures to wrap functions. They are the foundation of frameworks (Flask routes, Django views, pytest fixtures).
Key ideas

nonlocal lets inner function modify (not just read) an enclosing variable.

@wraps(func) from functools preserves __name__, __doc__, __annotations__.

Decorator with arguments needs 3 levels of nesting: decorator factory → decorator → wrapper.

Stacked decorators apply bottom-up: @A @B def f → f = A(B(f))

@wraps(func) is not optional — without it, your decorated function loses its __name__ and __doc__. This breaks introspection, pytest, sphinx docs, and any tool that reads function metadata.
Code example
from functools import wraps

# decorator with arguments
def retry(times=3, exceptions=(Exception,)):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(times):
                try: return func(*args, **kwargs)
                except exceptions as e:
                    if attempt==times-1: raise
        return wrapper
    return decorator

@retry(times=3, exceptions=(IOError,))
def fetch_data(url): ...

# class-based decorator
class memoize:
    def __init__(self, fn):
        self.fn = fn; self.cache = {}
    def __call__(self, *args):
        if args not in self.cache:
            self.cache[args] = self.fn(*args)
        return self.cache[args]
Generators & iterators
Generators produce values lazily using yield. They implement the iterator protocol automatically. Use them for large datasets, infinite sequences, and memory-efficient pipelines.
Key ideas

A function with yield is a generator function. Calling it returns a generator object.

yield from iterable — delegate to sub-generator or any iterable.

send(value) — send a value INTO a running generator (coroutine pattern).

Generator expressions: (expr for x in it) — lazy, uses O(1) memory.

Generator pipelines process data one piece at a time — a 10GB file uses only ~1KB of memory. This is why generators are the backbone of data processing frameworks like pandas internals, SQLAlchemy, and Scrapy.
Code example
# generator pipeline
def read_chunks(filename, size=1024):
    with open(filename) as f:
        while chunk := f.read(size):
            yield chunk

def parse_lines(chunks):
    for chunk in chunks:
        yield from chunk.splitlines()

def filter_empty(lines):
    return (l for l in lines if l.strip())

# chain generators — no memory spike
chunks = read_chunks("big_file.txt")
lines  = parse_lines(chunks)
clean  = filter_empty(lines)
for line in clean: process(line)

OOP

Classes, objects & properties
Classes bundle state (attributes) and behaviour (methods). Python's OOP is flexible — everything is an object, including functions and classes themselves. Properties let you add validation without breaking the public API.
Key ideas

__init__ constructor. __new__ creates the instance (rarely overridden).

@property getter. @name.setter setter. @name.deleter deleter.

__slots__ restricts attributes and reduces memory usage by 40-50%.

@classmethod for alternative constructors. @staticmethod for utility functions.

__slots__ prevents arbitrary attribute assignment and reduces per-instance memory by ~40%. Use when creating many small objects. Downside: can't add arbitrary attributes, and multiple inheritance with __slots__ is tricky.
Code example
class Circle:
    __slots__ = ['_radius']   # saves memory

    def __init__(self, radius):
        self.radius = radius  # uses setter

    @property
    def radius(self): return self._radius
    @radius.setter
    def radius(self, v):
        if v < 0: raise ValueError("negative")
        self._radius = v

    @property
    def area(self):   # computed property
        import math
        return math.pi * self._radius**2

    @classmethod
    def unit(cls): return cls(1)

c = Circle.unit()
c.area  # 3.14159...
Inheritance, MRO & mixins
Python supports multiple inheritance. The Method Resolution Order (MRO) determines which method is called. Mixins are small classes that add specific behaviour without being stand-alone base classes.
Key ideas

super() follows MRO — don't hardcode the parent class name.

ClassName.__mro__ shows resolution order. C3 linearisation algorithm.

Mixins add behaviour without state — they should not have __init__.

isinstance(obj, (A, B)) checks multiple types at once.

Always call super().__init__() in cooperative multiple inheritance — it ensures the full MRO chain is initialised properly. Never call ParentClass.__init__(self) directly as it breaks MRO.
Code example
class LogMixin:
    def log(self, msg):
        print(f"[{self.__class__.__name__}] {msg}")

class JsonMixin:
    def to_json(self):
        import json
        return json.dumps(self.__dict__)

class Animal:
    def __init__(self, name):
        self.name = name

class Dog(LogMixin, JsonMixin, Animal):
    def __init__(self, name, breed):
        super().__init__(name)
        self.breed = breed
    def speak(self):
        self.log("Woof!")

rex = Dog("Rex", "Lab")
rex.to_json()
rex.speak()
Magic methods & operator overloading
Magic (dunder) methods define how objects interact with Python's built-in syntax and functions. Implementing them makes your objects feel native — they behave exactly like built-in types.
Key ideas

__repr__ for devs (unambiguous). __str__ for users (readable). print() calls __str__.

__eq__ enables ==. If you define __eq__, Python sets __hash__=None unless you define it.

__lt__, __le__, __gt__, __ge__ — or use @functools.total_ordering to fill in the gaps.

__getitem__, __setitem__, __len__, __contains__ — make objects behave like sequences/dicts.

@functools.total_ordering: define __eq__ and one of __lt__/__le__/__gt__/__ge__, and Python fills in the rest automatically. Saves writing all 6 comparison methods.
Code example
from functools import total_ordering

@total_ordering
class Version:
    def __init__(self, major, minor, patch):
        self.v = (major, minor, patch)
    def __repr__(self):
        return f"Version{self.v}"
    def __str__(self):
        return ".".join(map(str, self.v))
    def __eq__(self, o): return self.v == o.v
    def __lt__(self, o): return self.v < o.v
    def __hash__(self): return hash(self.v)

v1 = Version(1,2,3)
v2 = Version(2,0,0)
v1 < v2    # True
v1 <= v2   # True (from total_ordering)
sorted([v2, v1])   # [1.2.3, 2.0.0]
Abstraction & encapsulation
Abstraction hides complexity behind contracts (ABCs). Encapsulation controls access to internal state. Together they enforce clean interfaces and make code maintainable at scale.
Key ideas

@abstractmethod on any method forces subclasses to implement it.

You can also make abstract @property, @classmethod, @staticmethod.

__ name mangling makes it hard (not impossible) to access from outside.

Context managers (__enter__/__exit__) are a clean encapsulation of resource lifecycle.

@contextmanager is the easiest way to write context managers. The code before yield is __enter__, the code after (in finally) is __exit__. No need to write a full class with __enter__/__exit__.
Code example
from abc import ABC, abstractmethod

class Repository(ABC):
    @abstractmethod
    def get(self, id): ...
    @abstractmethod
    def save(self, entity): ...
    @abstractmethod
    def delete(self, id): ...

class UserRepo(Repository):
    def get(self, id):    return db.query(id)
    def save(self, u):    db.insert(u)
    def delete(self, id): db.delete(id)

# contextmanager decorator
from contextlib import contextmanager

@contextmanager
def managed_connection(url):
    conn = connect(url)
    try:
        yield conn
    finally:
        conn.close()

with managed_connection("db://...") as conn:
    conn.execute(query)

Dataclasses & Pydantic

@dataclass — typed classes without boilerplate
dataclasses auto-generate __init__, __repr__, __eq__ (and optionally __hash__, __lt__ etc.) from field annotations. They replace verbose class boilerplate and are the modern Pythonic way to define data-holding classes.
Key ideas

@dataclass — generates __init__, __repr__, __eq__ from annotated fields.

frozen=True — makes instances immutable and adds __hash__ (usable as dict key).

field(default_factory=list) — for mutable defaults (never use [] directly).

@dataclass(order=True) — also generates __lt__, __le__, __gt__, __ge__.

Use frozen=True when you want the dataclass to be hashable (usable as a dict key or set element). This is more efficient and safer than a regular class with __hash__ manually defined.
Code example
from dataclasses import dataclass, field, asdict, astuple

@dataclass
class Point:
    x: float
    y: float
    label: str = "origin"   # default

p = Point(1.0, 2.0)
print(p)     # Point(x=1.0, y=2.0, label='origin')
asdict(p)    # {'x':1.0, 'y':2.0, 'label':'origin'}

@dataclass(frozen=True, order=True)
class Version:
    major: int
    minor: int
    patch: int

# mutable default — use field()
@dataclass
class Team:
    name: str
    members: list[str] = field(default_factory=list)
    _id: int = field(default=0, repr=False)   # hide from repr
Pydantic — data validation & settings
Pydantic validates data at runtime using Python type hints. It's the standard for API data models (FastAPI), config management, and data parsing. It auto-converts types and provides clear error messages on invalid data.
Key ideas

Pydantic models inherit from BaseModel. Fields are annotated with type hints.

Pydantic validates AND coerces: '42' → 42 for an int field.

model.model_dump() → dict. model.model_json_schema() → JSON Schema.

@field_validator adds custom validation. @model_validator validates across fields.

Pydantic is the backbone of FastAPI. Every FastAPI endpoint's request body, response model, and query params are Pydantic models. Knowing Pydantic well is essential if you're interviewing for Python backend roles.
Code example
from pydantic import BaseModel, field_validator, EmailStr
from typing import Optional

class User(BaseModel):
    id:    int
    name:  str
    email: str
    age:   Optional[int] = None

    @field_validator('age')
    @classmethod
    def check_age(cls, v):
        if v is not None and v < 0:
            raise ValueError("age must be positive")
        return v

user = User(id="1", name="Alice", email="a@b.com")
# id coerced: "1" → 1
user.model_dump()
# {'id':1, 'name':'Alice', 'email':'a@b.com', 'age':None}
Pydantic settings & config management
pydantic-settings (BaseSettings) reads configuration from environment variables, .env files, and other sources. It's the standard way to manage app configuration in modern Python projects.
Key ideas

BaseSettings reads from environment variables automatically (case-insensitive).

Nested models: group related settings into sub-models.

model_config = ConfigDict(...) controls behaviour: strict types, aliases, extra fields.

Validation happens at import time — misconfigured apps fail fast at startup.

SecretStr hides sensitive values in logs and repr — settings.secret shows SecretStr('**********') not the actual value. Use get_secret_value() to access the raw string when needed.
Code example
from pydantic_settings import BaseSettings, SettingsConfigDict
from pydantic import SecretStr

class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file='.env',
        env_file_encoding='utf-8'
    )

    app_name: str = "MyApp"
    debug:    bool = False
    db_url:   str
    secret:   SecretStr   # hidden in repr
    workers:  int = 4

settings = Settings()   # reads .env + env vars

# .env file:
# DB_URL=postgresql://localhost/mydb
# SECRET=super-secret-key
# DEBUG=true

Asyncio

async/await — concurrency without threads
asyncio provides cooperative multitasking — one thread, many concurrent tasks. When a task waits for I/O (network, disk), it yields control so other tasks can run. Perfect for I/O-bound work: web servers, API calls, databases.
Key ideas

async def defines a coroutine — it doesn't run until awaited.

await expr pauses the current coroutine until expr completes. Only inside async def.

asyncio.run(main()) — entry point. Creates the event loop and runs the coroutine.

Use asyncio.gather() to run multiple coroutines concurrently.

asyncio is NOT parallelism — it's concurrency. One thread runs at a time, but while waiting for I/O another task runs. For CPU-bound work use multiprocessing or concurrent.futures.ProcessPoolExecutor instead.
Code example
import asyncio

async def fetch(url: str) -> str:
    await asyncio.sleep(1)   # simulate I/O
    return f"data from {url}"

async def main():
    # sequential — 3 seconds
    r1 = await fetch("url1")
    r2 = await fetch("url2")

    # concurrent — 1 second
    r1, r2, r3 = await asyncio.gather(
        fetch("url1"),
        fetch("url2"),
        fetch("url3")
    )
    print(r1, r2, r3)

asyncio.run(main())
async patterns — tasks, timeouts, queues
asyncio.Task wraps a coroutine and schedules it. Timeouts prevent hanging. Queues coordinate producers and consumers. These patterns are the building blocks of async applications.
Key ideas

asyncio.create_task(coro()) — schedule immediately, don't await yet.

asyncio.wait_for(coro(), timeout=5) — raises TimeoutError if too slow.

asyncio.Queue — async-safe producer/consumer queue.

async for / async with — for async iterators and context managers.

create_task() vs gather(): use create_task() when you want to start a task and do other work before awaiting it. Use gather() when you want to start multiple tasks and wait for all of them at once.
Code example
import asyncio

async def main():
    # create tasks — run concurrently
    t1 = asyncio.create_task(fetch("url1"))
    t2 = asyncio.create_task(fetch("url2"))
    r1 = await t1
    r2 = await t2

    # timeout
    try:
        result = await asyncio.wait_for(
            fetch("slow"), timeout=2.0)
    except asyncio.TimeoutError:
        print("Timed out!")

    # async context manager (aiohttp)
    async with aiohttp.ClientSession() as session:
        async with session.get(url) as resp:
            data = await resp.json()
aiohttp & httpx — async HTTP clients
For async HTTP requests, use aiohttp or httpx. Never use the requests library inside async code — it blocks the event loop and defeats the purpose of asyncio.
Key ideas

aiohttp.ClientSession — reuse session across requests (connection pooling).

httpx.AsyncClient — async-compatible, similar API to requests.

Always use async with for sessions — ensures proper cleanup.

asyncio.gather() — fire many requests concurrently and wait for all.

Fetching 100 URLs sequentially with requests takes ~100s (1s each). With asyncio + aiohttp gather() it takes ~1s. This is the core value of async programming for I/O-bound tasks.
Code example
import asyncio, aiohttp

async def fetch_all(urls: list[str]) -> list:
    async with aiohttp.ClientSession() as session:

        async def fetch_one(url):
            async with session.get(url) as r:
                return await r.json()

        return await asyncio.gather(
            *[fetch_one(u) for u in urls]
        )

# httpx (requests-like API)
import httpx

async def get_data(url):
    async with httpx.AsyncClient() as client:
        resp = await client.get(url)
        return resp.json()

results = asyncio.run(fetch_all([url1, url2, url3]))

Exceptions & Files

Exception handling — complete picture
Python exceptions form a hierarchy. Understanding the hierarchy lets you catch at the right level — not too broad, not too narrow. Custom exceptions with context and exception chaining are essential for production code.
Key ideas

Exception is the base for all non-system-exiting exceptions.

BaseException includes SystemExit, KeyboardInterrupt — never catch this.

raise X from Y — chains exceptions. raise X from None — suppresses chaining.

ExceptionGroup (Python 3.11+) — holds multiple exceptions at once. Handle with except*.

contextlib.suppress(ExcType) is cleaner than try/except/pass. Use it when you explicitly want to ignore a specific exception — it's self-documenting and avoids the empty except block anti-pattern.
Code example
# exception hierarchy
# BaseException
#   ├── SystemExit
#   ├── KeyboardInterrupt
#   └── Exception
#         ├── ValueError, TypeError
#         ├── OSError (IOError, FileNotFoundError)
#         └── RuntimeError, StopIteration

# contextlib.suppress — ignore specific errors
from contextlib import suppress
with suppress(FileNotFoundError):
    Path("missing.txt").unlink()

# Python 3.11+ ExceptionGroup
try:
    raise ExceptionGroup("errors", [
        ValueError("bad val"),
        TypeError("bad type")])
except* ValueError as eg:
    print("value errors:", eg.exceptions)
File I/O & pathlib
Python's pathlib.Path is the modern cross-platform API for file system operations. Combined with the with statement for safe file handling, it replaces all legacy os.path string-based code.
Key ideas

Path.read_text() / write_text() — read/write in one line, auto-closes.

Path.glob() / rglob() — pattern matching. rglob = recursive glob.

shutil module — copy, move, delete directories.

tempfile.TemporaryDirectory() — auto-cleanup temp dirs on exit.

Path.rglob('*.py') is equivalent to glob('**/*.py'). rglob is slightly more readable. Both return a generator — wrap in list() to materialise, or iterate directly to stay lazy.
Code example
from pathlib import Path
import shutil, tempfile

p = Path("data") / "output.txt"
p.parent.mkdir(parents=True, exist_ok=True)
p.write_text("hello", encoding="utf-8")
p.read_text()
p.stat().st_size      # file size in bytes

# find all .csv files recursively
csvs = list(Path(".").rglob("*.csv"))

# copy/move
shutil.copy(src, dst)
shutil.move(src, dst)
shutil.rmtree("dir_to_delete")

# temp directory
with tempfile.TemporaryDirectory() as tmp:
    p = Path(tmp) / "work.txt"
    p.write_text("temp data")
    process(p)
# tmp auto-deleted here

Modules & Stdlib

Modules, packages & imports
Python's import system lets you organise code into modules and packages. Understanding relative vs absolute imports, __init__.py, and __all__ is essential for building real-world packages.
Key ideas

import package.module — absolute import (always preferred).

from . import module — relative import (inside a package only).

__all__ in a module — controls what 'from module import *' exports.

importlib.import_module(name) — dynamic import at runtime.

Never use 'from module import *' in production code — it pollutes the namespace and makes it impossible to know where a name came from. Only use it in interactive sessions or if the module explicitly defines __all__.
Code example
# mypackage/
#   __init__.py   ← makes it a package
#   utils.py
#   models/
#     __init__.py
#     user.py

# absolute import
from mypackage.models.user import User

# relative import (inside package)
from ..utils import helper  # go up 2 levels
from .models import User

# __all__ — controls star imports
# in utils.py:
__all__ = ['helper', 'format_date']

# dynamic import
import importlib
mod = importlib.import_module("json")
mod.dumps({"key": "val"})
Standard library essentials
Python's standard library is enormous. These are the modules that appear in almost every real project — know them well. The stdlib philosophy: batteries included.
Key ideas

collections — deque, Counter, defaultdict, namedtuple, OrderedDict, ChainMap.

itertools — chain, islice, groupby, combinations, permutations, product, cycle.

functools — wraps, partial, reduce, lru_cache, cache, total_ordering, cached_property.

contextlib — contextmanager, suppress, nullcontext, ExitStack.

@cached_property computes the value once and stores it as an instance attribute. Subsequent accesses skip the method entirely — it's O(1) after the first call. Perfect for expensive computed properties.
Code example
# collections.ChainMap — layered config
from collections import ChainMap
defaults = {"color": "red", "size": 10}
user_cfg = {"size": 20}
config = ChainMap(user_cfg, defaults)
config["color"]  # "red" (from defaults)
config["size"]   # 20   (from user_cfg)

# contextlib.ExitStack — dynamic contexts
from contextlib import ExitStack
files = ["a.txt", "b.txt", "c.txt"]
with ExitStack() as stack:
    handles = [stack.enter_context(open(f))
               for f in files]
    # all files auto-closed on exit

# functools.cached_property
from functools import cached_property
class DataSet:
    @cached_property
    def stats(self):
        return expensive_compute(self.data)

Logging

Logging — setup & best practices
Structured logging is the foundation of observable production systems. Python's logging module is highly configurable — handlers, formatters, filters, and log levels let you control exactly what gets recorded where.
Key ideas

Use getLogger(__name__) in every module — creates a hierarchy matching package structure.

Set level on the root logger or specific loggers — child loggers inherit parent level.

propagate=False stops records from going to ancestor loggers.

Never use print() in production — can't be filtered, redirected, or levelled.

Use lazy string formatting: logger.debug('Value: %s', x) NOT logger.debug(f'Value: {x}'). With f-strings the string is always built even if the log level suppresses output. % formatting is deferred.
Code example
import logging
import sys

# configure once at app entry point
logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s %(name)s %(levelname)s %(message)s',
    datefmt='%Y-%m-%dT%H:%M:%S',
    handlers=[
        logging.StreamHandler(sys.stdout),
        logging.FileHandler('app.log')
    ]
)

# in each module
logger = logging.getLogger(__name__)

logger.debug("Processing item %s", item_id)  # lazy %s
logger.info("User %s logged in", username)
logger.warning("Retrying: attempt %d", n)
logger.error("DB query failed", exc_info=True)
Structured logging & rotating handlers
Structured logging outputs JSON instead of text — makes logs machine-readable and searchable in tools like Elasticsearch, Splunk, or Datadog. RotatingFileHandler prevents log files from growing indefinitely.
Key ideas

python-json-logger — formats log records as JSON (pip install python-json-logger).

RotatingFileHandler — rotates when file reaches maxBytes.

TimedRotatingFileHandler — rotates daily/hourly regardless of size.

logging.Filter — add custom logic (e.g. filter by user_id, request_id).

Structured (JSON) logs are essential for production. Text logs require regex to parse; JSON logs can be queried with jq, indexed in Elasticsearch, and filtered in CloudWatch — all without custom parsers.
Code example
import logging
from logging.handlers import RotatingFileHandler
from pythonjsonlogger import jsonlogger

def setup_logging():
    logger = logging.getLogger()
    logger.setLevel(logging.INFO)

    # JSON formatter
    handler = RotatingFileHandler(
        'app.log',
        maxBytes=10_000_000,   # 10 MB
        backupCount=5
    )
    fmt = jsonlogger.JsonFormatter(
        '%(asctime)s %(name)s %(levelname)s %(message)s'
    )
    handler.setFormatter(fmt)
    logger.addHandler(handler)

# Output (one JSON line per record):
# {"asctime":"2024-01-15","name":"app.api",
#  "levelname":"INFO","message":"Request received"}
Logging with context — filters & adapters
Real-world logging needs context — request IDs, user IDs, trace IDs. Python's LoggerAdapter and Filter let you attach context to every log record without passing it explicitly to every log call.
Key ideas

LoggerAdapter — wraps a logger and adds extra context to every record.

logging.Filter — subclass and implement filter(record) to add/modify/drop records.

contextvars.ContextVar — store per-request context in async code.

logging.config.dictConfig() — configure logging from a dict or YAML file.

ContextVar is the correct way to store per-request state in async code. Unlike threading.local, it's isolated per async Task — different concurrent requests get their own context without interference.
Code example
import logging
from contextvars import ContextVar

request_id: ContextVar[str] = ContextVar('request_id', default='—')

class RequestFilter(logging.Filter):
    def filter(self, record):
        record.request_id = request_id.get()
        return True

logger = logging.getLogger(__name__)
logger.addFilter(RequestFilter())

# in a request handler
async def handle_request(req_id: str):
    token = request_id.set(req_id)
    try:
        logger.info("Processing")
        # every log line now has request_id
    finally:
        request_id.reset(token)

NumPy & Pandas

NumPy — arrays & operations
NumPy's ndarray is the foundation of scientific Python. Operations run in compiled C/Fortran — 10-100x faster than Python loops. Understanding broadcasting, views vs copies, and axis is essential for data roles.
Key ideas

Broadcasting: NumPy automatically expands smaller arrays to match larger ones in arithmetic.

Views vs copies: slices are views (shared memory). Use .copy() for independence.

axis=0 = down rows (column-wise result). axis=1 = across columns (row-wise result).

np.vectorize(fn) — apply Python function element-wise (not truly fast — use built-ins).

np.where(condition, x, y) is the vectorised if-else. np.select([c1,c2,c3], [v1,v2,v3], default) is the vectorised elif chain. Both are essential for creating derived columns without loops.
Code example
import numpy as np

# broadcasting example
m = np.array([[1,2,3],[4,5,6]])  # (2,3)
v = np.array([10,20,30])           # (3,)
m + v   # adds v to each row → (2,3)

# view vs copy
a = np.arange(6).reshape(2,3)
b = a[0]     # view — shared memory
c = a[0].copy()  # independent copy
b[0] = 99    # modifies a too!
c[0] = 99    # does NOT modify a

# useful operations
np.unique(arr)          # unique values
np.clip(arr, 0, 1)     # clamp values
np.concatenate([a,b])   # join arrays
np.stack([a,b], axis=0) # new dimension
Pandas — DataFrame operations
Pandas is built on NumPy and adds labels, missing data handling, and SQL-like operations. The key mental model: think in columns, not rows. Vectorised column operations are always faster than row-by-row apply.
Key ideas

df.assign(col=expr) — add columns without mutation (method-chaining friendly).

df.pipe(fn) — apply a function to the whole DataFrame (great for pipelines).

pd.cut() / pd.qcut() — bin continuous data into categories.

df.explode(col) — expand list-valued column into multiple rows.

Method chaining with .pipe() and .assign() is the pandas equivalent of a Unix pipeline — each step is clear and the whole transformation is readable in one block without intermediate variables.
Code example
import pandas as pd

# method chaining — clean pipeline
result = (
    pd.read_csv("sales.csv")
      .dropna(subset=["price","qty"])
      .assign(revenue=lambda df: df.price * df.qty)
      .query("revenue > 1000")
      .groupby("category")
      .agg(total=("revenue","sum"))
      .reset_index()
      .sort_values("total", ascending=False)
)

# pd.cut — bin into categories
df["age_group"] = pd.cut(df["age"],
    bins=[0,18,35,60,100],
    labels=["child","young","mid","senior"]
)

# explode — list column to rows
df["tags"] = [["a","b"], ["c"], ["a","d"]]
df.explode("tags")   # 5 rows from 3
Reading data from various sources
Pandas can read from CSV, Excel, JSON, SQL, Parquet, HTML, and APIs. Understanding which format to choose and how to handle large files efficiently is important for data engineering interviews.
Key ideas

pd.read_csv(chunksize=N) — reads large CSVs in chunks without loading all to memory.

pd.read_parquet() — columnar format, fast and compressed. Preferred for big data.

pd.read_sql(query, conn) — execute SQL and return DataFrame.

df.to_parquet() / to_csv() / to_excel() — write to various formats.

Parquet is the best format for storing large DataFrames. It is 5-10x smaller than CSV (columnar + compression), reads 10-100x faster (columnar skipping), and preserves dtypes exactly. Always prefer Parquet over CSV for data pipelines.
Code example
import pandas as pd
import sqlalchemy

# CSV in chunks (large files)
chunks = pd.read_csv("big.csv", chunksize=10_000)
result = pd.concat(
    [chunk[chunk.value > 0] for chunk in chunks]
)

# Parquet — fast columnar format
df.to_parquet("data.parquet", index=False)
df = pd.read_parquet("data.parquet")

# SQL database
engine = sqlalchemy.create_engine("sqlite:///db.sqlite")
df = pd.read_sql("SELECT * FROM sales WHERE year=2024", engine)
df.to_sql("results", engine, if_exists="replace")

# JSON / Excel
df = pd.read_json("data.json")
df = pd.read_excel("report.xlsx", sheet_name="Q1")
